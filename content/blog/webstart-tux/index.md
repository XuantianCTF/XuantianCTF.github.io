## 先泼盆冷水

很多人学 CTF Web，路径是这样的：做几道 SQL 注入，背几个 XSS payload，会用 Burp 抓包——然后觉得自己"入门了"。

这叫**应试技巧**，不叫**安全能力**。

出题人换个姿势你就不会了。因为你只记住了"怎么解"，没理解"为什么存在"。

CTF Web 真正的门槛，不是攻击手法的数量，而是你对 Web 系统底层机制的理解深度。

---

## 先搞懂 Web 是怎么跑起来的

在想怎么"打"之前，先搞清楚它是怎么"活"的。

### HTTP 协议

会用 Burp 抓包不算懂 HTTP。你得能说清楚：DNS 解析流程、TCP/TLS 握手过程、请求头各字段含义、Cookie/Session 的识别机制、301 与 302 的区别、Keep-Alive 的作用、HTTP/1.1 到 HTTP/3 的演进。

这些看起来枯燥，却是理解所有 Web 漏洞的根基。

### 认证与授权

会 `document.cookie` 弹 cookie 只是初学者水平。你需要搞清楚：Cookie 各属性（HttpOnly、Secure、SameSite）的作用、Session 的存储机制（内存/Redis/DB）、JWT 验签流程及 `none` 算法绕过原理、OAuth 2.0 授权流程及 state 参数的意义。

---

## 前端知识是必修课，不是加分项

很多 Web 选手对前端一知半解，觉得前端就是写页面。这个认知本身就是漏洞。

- **DOM**：XSS 的本质是攻击者能操作 DOM。不理解 DOM 结构，就无法理解 DOM-based XSS。
- **JS 执行模型**：事件循环、闭包、原型链——不是让你写业务代码，而是让你理解前端代码的执行逻辑。
- **CORS / CSP**：浏览器安全模型的核心。CORS 配置错误直接导致数据泄露，CSP 配置不当形同虚设。
- **框架特性**：React 的 JSX、Vue 的 `v-html`、Angular 的模板注入——每个框架都有独特的安全陷阱。
- **浏览器安全机制**：同源策略、Sandbox、Service Worker——了解防御才能找到绕过路径。

---

## 后端知识是核心

Web 安全的本质是**信任边界问题**。不理解服务端如何处理请求，就不可能找到信任边界上的漏洞。

### 语言

CTF 最常见的是 PHP，但不能只会 `<?php system($_GET['cmd']); ?>`：

- PHP 类型松散比较（`==` vs `===`）的绕过逻辑
- PHP 反序列化机制：`__wakeup`、`__destruct`、`__toString` 的配合
- 触类旁通：Node.js 原型链污染、Python pickle 反序列化、Java 反序列化链

### 数据库

会用 `sqlmap` 不算懂 SQL 注入。你需要理解：SQL 解析执行流程、各数据库（MySQL/PostgreSQL/SQLite/Oracle）语法差异、预编译语句的防注入原理（远不止"用了就安全"）、二次注入/堆叠注入/宽字节注入的成因、ORM 框架的安全边界。

### 服务器与中间件

Nginx、Apache、IIS 的配置差异、路径解析规则、请求大小限制，都可能成为绕过 WAF 或访问控制的切入点。容器化时代，Docker 逃逸和 Kubernetes 配置安全也越来越多地出现在 CTF 中。

---

## 底层细节决定上限

高分题考的都是细节：

- **编码绕过**：URL 编码、HTML 实体编码、Unicode 编码的组合——为什么 `%27` 有时能绕过过滤，有时不能？
- **解析差异**：浏览器与 Web 服务器对 `/./`、`/;`、`%2f` 的解析可能完全不同。
- **竞态条件**：并发请求的时间差如何被利用？
- **反序列化**：不同语言和框架的反序列化机制各异，如何构造完整利用链？
- **SSTI**：Jinja2、Twig、Smarty 的沙箱绕过方式完全不同。

这些靠刷题数量堆不出来，需要读源码、读文档、读 RFC。

---

道理都懂了，怎么落地？

## 学习路径

### 1. 打牢基础

别急着刷题。HTTP 协议、Web 服务器原理、数据库基础——先搞清楚。推荐《HTTP 权威指南》（*HTTP: The Definitive Guide*），厚但有用。

### 2. 学一门后端语言

PHP、Python、Node.js 都行，但要真正写过 Web 应用，不是 CTF 脚本。亲手搭过服务、写过接口、处理过认证，才能真正理解攻击面。

### 3. 读原理，不只是 payload

遇到新漏洞，先搞清楚：成因是什么、利用条件有哪些、防御方法为何有效、同类漏洞在不同环境下的差异。

### 4. 搭自己的靶场

DVWA、Pikachu、sqli-labs——自己改配置、加防护、想办法绕过。比单纯做题有效得多。

### 5. 读优秀 WriteUp

看高手怎么思考：如何分析题目、定位漏洞、构造利用链。好的 WriteUp 学的是思路，不是答案。

### 6. 关注真实漏洞

CVE、CNVD 中有详细分析报告的真实漏洞，复杂度远超 CTF 题目，能帮你建立更完整的安全观。

### 7. 提升代码和工程能力

很多选手**会打不会写**。Exploit 脚本能跑，但写不出干净、可复用的工具。

- **脚本 ≠ 代码**：临时脚本变量乱起、无异常处理、hardcode 满天飞——两周后自己都看不懂。常用脚本应整理成正经工具，加参数解析、日志、错误处理。
- **读源码**：Web 题的本质是读源码找漏洞。平时不读代码，比赛时面对几百行代码会无从下手。多读开源项目，培养快速定位关键逻辑的能力。
- **理解开发框架**：搞懂 Laravel 反序列化漏洞，得先理解 Laravel 架构；搞懂 Spring SpEL 注入，得先知道 Spring 的请求处理流程。安全与开发是硬币两面，不懂开发就看不到漏洞全貌。
- **版本控制**：用 Git 管理 WriteUp、脚本和笔记。这是建立知识体系的基础，不是"程序员的事"。

CTF 打到最后拼的不是 payload 数量，而是**工程底子**。能写代码、能读源码、能搭环境的人，在任何安全领域都不会差。

---

## 总结

CTF Web 的学习，本质上是学**如何理解一个复杂系统**，而不是学**如何套模板做题**。

底子扎实的人，遇到没见过的题也能分析。只会套模板的人，换个场景就抓瞎。

别再问"CTF Web 要刷多少题才能拿奖"——先把底子打扎实，题自然就会做了。
